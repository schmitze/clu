## Schritt 8 — Erste Session starten

Nach erfolgreichem Heartbeat:

```bash
clu                # workspace mode — Pre-Session-Curator sollte instant
                   # durchgehen ("Nothing to consolidate")
                   # und ggf. den Hinweis auf memory-review-pending.md zeigen
```

Wenn der Pre-Session-Curator was findet, gut — er konsolidiert die letzte
Session bevor das Memory geladen wird.

### Was beim ersten `clu <projekt>` neu ist

- **Auto-Resume-Check** vor dem Memory-Load (~10–15s, Claude Haiku 4.5
  über die Subscription): „war die letzte Session in diesem Projekt
  abgeschlossen oder mid-flow?". Bei mid-flow → `claude --continue`,
  voller Thread, kein Memory-Inject. Bei abgeschlossen oder bei Timeout →
  normaler Flow weiter. Mi sieht eine Zeile „🔍 Checking last session for
  mid-flow continuation…".
- **Override `clu --fresh <projekt>`** überspringt den Auto-Check und
  startet immer fresh.
- **Override `clu resume <projekt>`** überspringt den Check ebenfalls,
  aber resumed bedingungslos.

Wenn der Auto-Resume-Check beim ersten Test stört (z.B. ist mi-geek
am ein in einer langen Session und der Check wird teuer / falsch
entscheidet), kann er mit `--fresh` erstmal abgeschaltet werden.

### Strukturierter `/compact`

`shared/core-prompt.md` weist den Live-Agent jetzt an, bei `/compact`
ohne Fokus-Argument das OpenCode-Schema zu nutzen (Goal / Constraints /
Progress / Key Decisions / Next Steps / Critical Context). Plus: bei
schweren Sessions schlägt der Agent proaktiv einen Compact mit Fokus
vor, statt eigenmächtig zu komprimieren. Plus: nach `Read` von Files
> 500 Zeilen schreibt der Agent ein 3-5-Sätze-Digest in eigenen Worten
statt File-Content zu re-quoten. Das ist Workflow-Disziplin emuliert
das was OpenCode automatisch im Hintergrund macht — siehe
`docs/CLU-DOCUMENTATION.md` Abschnitt „Lange Sessions / Context-Management".

## Was wenn was schiefgeht

**Heartbeat hängt oder timed out:**
```bash
# laufenden Prozess finden
pgrep -af heartbeat\.sh

# wenn länger als 40 min: hart killen, Logs anschauen
kill <pid>
journalctl --user -u clu-heartbeat.service --since "1 hour ago" --no-pager
```

**Memory-Repo Push schlägt fehl:**
- Heartbeat-Task-9 macht `pull --rebase` vor `push`. Wenn das auch
  scheitert (zu viel Divergenz), stoppt der Heartbeat den Sync und
  loggt eine Warnung. Manuell: `cd ~/repos/clu-memory && git pull --rebase`,
  Konflikte lösen, push.

**Pre-Session-Curator hängt:**
- Hat 60s Timeout im Launcher. Wenn er regelmäßig timed out: Backlog
  zu groß, einmalig manuell laufen:
  `~/.clu/tools/curator/curator.py run --limit 50`

**`shared/agent/` taucht wieder auf:**
- Heißt der Heartbeat-Code (`~/.clu/heartbeat.sh`) ist noch alt. Nochmal
  Schritt 3 ausführen.

## Konflikt-Strategie für Memory-Sync (allgemeine Regel)

Mi hat festgelegt: bei Memory-Konflikten zwischen den Maschinen gewinnt
**framework**. Ausnahme: **kosi-webxr** — da gewinnt **mi-geek**, weil
KoSi-Arbeit dort gemacht wird.

Konkret beim merge:

```bash
# Für alle Files außer kosi-webxr: framework-Stand
git checkout --theirs -- $(git diff --name-only --diff-filter=U | grep -v kosi-webxr)

# Für kosi-webxr: mi-geek-Stand (=HEAD lokal auf mi-geek)
git checkout --ours -- $(git diff --name-only --diff-filter=U | grep kosi-webxr)
```

Bei strukturellen Konflikten (Numbering-Race in DEC/FND/LRN — gleiche
Nummer auf beiden Seiten unterschiedlich befüllt): den später entstandenen
Eintrag manuell renumbern und in der jeweils anderen Datei den Inhalt
behalten.
